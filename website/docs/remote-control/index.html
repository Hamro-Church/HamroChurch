Remote Control

Share control with another operator without losing service flow.

Hamro Church remote tools are useful when one person presents slides while another manages stage, stream, or supporting tasks over the same church network.

Best for

Assistant operators and technical teams

Use remote tools only after the main presentation machine is stable and accessible on your local network.

Remote basics

Keep the main presentation computer and remote devices on the same trusted network. Use passwords or restricted access where appropriate and avoid exposing control interfaces beyond the local environment unless you have a specific, secure design.

Controller view

The controller view is useful for a second operator who needs awareness of the current service state without taking over the full editor. Keep responsibilities clear so two people are not fighting the same control path.

API access

API access is helpful when Hamro Church needs to integrate with another tool or automation surface. Treat the API as an operational tool, not a casual shortcut, and document any commands your team depends on.

Operational advice: Before adding automation, verify that your manual workflow is already reliable. Automation should reduce load, not hide a confusing service process.

Bitfocus Companion

Bitfocus Companion is a strong option when your church already uses button-driven control surfaces. Map only the buttons your operators actually need and rehearse every action in the exact service context.

REST listener planning

If your church plans to expand into a broader automation or API-driven control model, reserve time for testing, permissions, and network design. Even when a REST listener is not yet in daily use, the workflow should be documented before it becomes critical.
